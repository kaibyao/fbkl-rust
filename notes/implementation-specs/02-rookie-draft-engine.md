# Spec 02 — Rookie Draft Engine

**Rules ref:** §7, §1.4 · **Status:** 🔴 not built · **Priority:** P0

## Summary

The entity write-path for rookie selections already exists — `import-data` replays historical
drafts through `rookie_draft_selection_queries::{insert_used_rookie_draft_selection,
insert_skipped_rookie_draft_selection}`, `contract::Model::new_contract_from_rookie_draft`, and the
`RookieDraftSelection` transaction + `AddViaRookieDraft` team_update. What is **missing is the live
draft**: computing draft order from standings + a lottery, presenting picks on-the-clock, and the
make/pass/re-draft-ban logic that a live draft needs (the importer cheats — it reads the answer from
the CSV and back-fills `order`/`Skipped` rows). This spec adds a `logic/src/rookie_draft/` module
that produces the same entity rows the importer does, but driven by live input instead of CSV.

Key facts to honor (§7):
- 5 rounds (`DRAFT_PICK_ROUNDS` = 5). Picks already generated by
  `logic::draft_picks::generate_future_draft_picks` at `end_of_season_year + 2`.
- Round 1 order = 6-team lottery (§7.2.4 balls 6/5/4/3/2/1) then playoff teams by playoff finish.
- Rounds 2–5 = reverse of the ≈2/3-season standings for non-playoff teams, then reverse playoff
  finish; **same order every round** (§7.2.1).
- Rookies drafted as RD contracts (`ContractKind::RookieDevelopment`) at the round's fixed salary
  ($4/$3/$2/$1/$1, §7.4.1).
- An owner may pass any/all picks (§7.3.1); passing requires nothing. Making a pick requires an open
  roster slot but **not** cap space (§7.3.2).
- Re-draft ban: a player dropped during *this* draft may not be drafted in the same draft (§7.3.4).

## Backend

### Data inputs (standings snapshot @ ~2/3 season, final standings, playoff finish) — where stored, how ingested

The draft engine **consumes** three derived inputs it cannot compute itself (see spec 12 §"Derived
data FBKL MUST ingest" and spec 10):

1. **≈2/3-season standings snapshot** — ranks all 12 teams. Drives lottery odds (§7.2.2) and the
   reverse order for rounds 2–5 (§7.2.1). The "exact week" is set before the season (§7.2.3).
2. **Final regular-season standings** — used only as the tie-break among playoff teams knocked out
   in the same round (§7.2.1) and among same-ball lottery edge cases (see open questions).
3. **Playoff finish** — orders the playoff teams in round 1 (best finish drafts last) and rounds 2–5
   (reverse: best finish drafts first within the playoff block).

**Storage** — add a new entity table `league_team_season_standing` (one row per team per
end_of_season_year), columns:
- `league_id`, `team_id`, `end_of_season_year`
- `regular_season_rank: i16` (final standings, 1 = best)
- `mid_season_rank: i16` (the ≈2/3 snapshot, 1 = best)
- `made_playoffs: bool`
- `playoff_finish: Option<i16>` (1 = champion … 6 = first-round loser; NULL for non-playoff teams)
- `created_at`/`updated_at`

This keeps the snapshot immutable (the rules freeze it at a set week) rather than recomputing from a
live feed. **Ingestion**: manual commissioner entry via a GraphQL mutation (cross-ref spec 06) plus
an optional Fantrax sync later (spec 12 marks this as required-but-external-sourced). The draft
engine reads it via a new `league_team_season_standing_queries::find_standings_for_league_season`.
Do **not** redesign eligibility here (spec 10) — the engine consumes an eligible-player pool.

### constants/ (add round→salary map)

Add to `constants/src/league_rules/config_settings.rs` (per `logic/CLAUDE.md` convention #
"rule values live in constants, never hardcode in logic"):

```rust
/// Fixed Rookie-Development contract salary for a rookie drafted in the given round (§7.4.1).
/// Index 0 = round 1. Rounds 1–5 → $4/$3/$2/$1/$1.
pub static ROOKIE_DRAFT_ROUND_SALARIES: [i16; 5] = [4, 3, 2, 1, 1];
```

Add a helper `pub fn rookie_draft_salary_for_round(round: i16) -> i16` in the same crate so the
1-based round maps cleanly and out-of-range panics loudly rather than silently indexing.

Also add lottery ball counts as a constant so they're not hardcoded:
```rust
/// Lottery balls per non-playoff seed, worst → best (§7.2.4). 6 seeds.
pub static ROOKIE_DRAFT_LOTTERY_BALLS: [u32; 6] = [6, 5, 4, 3, 2, 1];
```

### entity/ (lottery result storage, draft order, selection state — what new columns/tables)

The existing `rookie_draft_selection` table already models selection state well: `order` (overall
pick number, 1-based), `status` (`Unused`/`PlayerSelected`/`Skipped`), `contract_id`, `draft_pick_id`,
`league_id`. **Reuse it** — but today `order` is back-filled by the importer. For the live draft we
need the *full ordered slate* of picks up front so the UI can show who is on the clock. Two additions:

1. **Draft order / on-the-clock**: add `current_owner_team_id: i64` to `rookie_draft_selection`
   (denormalized from `draft_pick.current_owner_team_id` at the time the slate is built — picks
   acquired via trade are owned by the *current* owner, see open questions). Pre-create one
   `rookie_draft_selection` row per `draft_pick` in the season's slate with `status = Unused` and the
   computed `order`, instead of inserting them lazily as the importer does. The team "on the clock"
   is the `current_owner_team_id` of the lowest-`order` `Unused` selection.

2. **Lottery audit**: add table `rookie_draft_lottery` (one row per league/season) +
   `rookie_draft_lottery_pick` (one row per the 6 drawn first-round slots), storing for each drawn
   slot: `pick_number: i16` (1–6), `team_id`, `balls_held: i16`, and a `rng_seed: i64` / `rng_log`
   on the parent for auditability (see open questions on determinism). The first-round playoff-team
   order (picks 7–12) is deterministic from `playoff_finish` and does not need lottery storage.

New entity query module `rookie_draft_selection_queries` additions:
- `build_draft_slate(league_id, end_of_season_year, ordered_draft_pick_ids, db)` — bulk-insert the
  `Unused` selection rows in `order`.
- `get_on_the_clock_selection(league_id, end_of_season_year, db)` — lowest-`order` `Unused` row.
- `get_selections_for_draft(league_id, end_of_season_year, db)` — full slate for the board.
- `find_players_dropped_during_draft(league_id, end_of_season_year, db)` — for the re-draft ban
  (query `transaction` rows of kind `TeamUpdateDropContract` whose effective_date falls within the
  draft window; resolve to `player_id`/`league_player_id`).

### logic/ (draft order computation, lottery RNG, make-pick, pass-pick, re-draft ban check)

New module `logic/src/rookie_draft/` (register in `logic/src/lib.rs`), mirroring the conventions in
`logic/CLAUDE.md` (transaction + team_update on every state change; validate before mutating; wrap
multi-step mutations in `db.begin()`/`commit()`; delegate persistence to `entity/queries`).

**`draft_order::compute_draft_order`** — pure-ish function (reads standings, returns the ordered list
of `(round, draft_pick_id, current_owner_team_id)` tuples):
- Round 1: lottery result (picks 1–6 = the 6 non-playoff teams in drawn order) then playoff teams
  ordered by `playoff_finish` descending (worst playoff finish drafts at pick 7, champion at pick 12).
  Same-round playoff knockouts tie-broken by `regular_season_rank` (§7.2.1).
- Rounds 2–5: non-playoff teams in **reverse** `mid_season_rank` (worst drafts first) then playoff
  teams in reverse `playoff_finish`. Identical order each round (§7.2.1). Resolve each slot to the
  `draft_pick` whose `original_owner_team_id` matches the team for that round, then carry its
  `current_owner_team_id` so traded picks land with the acquirer.

**`lottery::run_lottery`** — draws picks 1→6 sequentially (§7.2.5):
- Build a weighted pool from `ROOKIE_DRAFT_LOTTERY_BALLS` keyed by `mid_season_rank` of the 6
  non-playoff teams (worst = 6 balls).
- Use a **seeded** RNG (`rand::rngs::StdRng::seed_from_u64(seed)`) so the draw is reproducible and
  auditable; persist `seed` + the per-draw ball counts to `rookie_draft_lottery`.
- After each draw, remove the winning team and re-draw from the remaining pool for the next pick.
- Returns ordered `[team_id; 6]`; persisted via the lottery tables. Must run after `SeasonEnd` and
  before `PreseasonRookieDraftStart` (§7.2.2).

**`start_rookie_draft`** — orchestrator: assert the `PreseasonRookieDraftStart` deadline exists,
`run_lottery`, `compute_draft_order`, then `build_draft_slate` to persist the `Unused` selection rows.

**`make_pick`** — selects a player with an `Unused` selection (must be the on-the-clock one):
1. **Validate before mutating** (convention #4): selection is the on-the-clock row; the player is in
   the eligible pool (spec 10); **re-draft ban** (§7.3.4) — player not in
   `find_players_dropped_during_draft`; the picking team (`current_owner_team_id`) has an open RD
   roster slot (`REGULAR_SEASON_ROOKIE_DEVELOPMENT_CONTRACTS_PER_ROSTER_LIMIT` style check — but note
   §7.3.2 only requires *roster* room, not cap, since RD doesn't count against cap).
2. Create the RD contract via `contract::Model::new_contract_from_rookie_draft(..., salary, ...)`
   where `salary = rookie_draft_salary_for_round(round)` → `contract_queries::create_new_contract`.
3. Flip the selection to `PlayerSelected` with `contract_id` set (reuse the
   `from_rookie_contract`/`insert_used_rookie_draft_selection` path).
4. Insert a `RookieDraftSelection` transaction (`transaction::Model::new_rookie_draft_selection_transaction`)
   and a `Done` `team_update` with `ContractUpdateType::AddViaRookieDraft` — exactly as
   `seasonal_rookie_selection_transactions::process_rookie_selected` does. Salary/cap unchanged
   (RD off-cap).

**`pass_pick`** — flips the on-the-clock selection to `Skipped`, inserts a `RookieDraftSelection`
transaction, and **no** team_update (no asset changed hands — mirror `mark_draft_pick_as_skipped`).

**`re_draft_ban_check`** — helper used by `make_pick`; surfaced separately so the GraphQL eligible-pool
query (below) can grey out banned players. Drops during the draft are detected via
`TeamUpdateDropContract` transactions within the draft window; §7.3.3 allows drops during the draft.

All of make/pass should run inside one DB transaction and re-assert "still on the clock" to guard
against concurrent picks (a draft is naturally serialized but two clients could race).

### GraphQL (cross-ref spec 06)

Resolvers live in `server/` and delegate to `logic::rookie_draft` (server has no business logic).
Cross-ref spec 06 for the broader read/write surface; this domain needs:
- **Queries**: `rookieDraft(leagueId, endOfSeasonYear)` → board (full slate w/ status, team-on-clock,
  resolved player + salary), `rookieDraftLottery` → audited lottery result, `eligibleRookiePool` →
  eligible players with a `banned: Boolean` flag (re-draft ban + already-drafted).
- **Mutations**: `runRookieDraftLottery`, `startRookieDraft`, `makeRookieDraftPick(selectionId,
  playerId)`, `passRookieDraftPick(selectionId)`, plus a commissioner mutation to ingest standings
  (`saveLeagueTeamSeasonStandings`).

## Frontend (Next.js + MUI v7 + urql)

- **Draft board**: grid of all 60 picks (5 rounds × 12), grouped by round, each cell showing pick #,
  owning team, and the selected player (or "Pass"/empty). Highlight the on-the-clock cell.
- **Lottery reveal**: an animated/stepped reveal of picks 1→6 driven by the audited
  `rookieDraftLottery` result (it's pre-computed server-side and seeded — the UI only *reveals* the
  stored order, it does not draw). Show each team's ball count and odds.
- **On-the-clock pick UI**: visible only to the owner of the current pick (or commissioner) — search
  the eligible pool, confirm a player, submit `makeRookieDraftPick`. Cap is irrelevant; show RD-slot
  count vs. limit instead.
- **Pass**: a clearly-separated "Pass pick" action (§7.3.1) that submits `passRookieDraftPick`.
- **Eligible-player pool browser**: paginated/searchable list from `eligibleRookiePool`; visually
  disable players that are `banned` (re-draft ban or already drafted), with a tooltip explaining why.
- Live updates via urql subscription or polling so all owners see picks as they happen.

## Edge cases & open questions

- **Lottery determinism / auditability**: persist the RNG `seed` and per-draw ball counts so the draw
  can be independently re-verified. Open question: should the seed be committed *before* the draw
  (commit-reveal) so owners trust it wasn't re-rolled? Recommend yes — store seed at lottery-row
  creation, reveal after.
- **Tie-breaking**: §7.2.1 breaks same-round playoff-knockout ties by regular-season record. Confirm
  `regular_season_rank` is sufficient (the rules' general tie-break in §1.4.1 is category wins — but
  that's resolved upstream into the final standings rank we ingest, so rank alone should suffice).
  Also: can two non-playoff teams tie in `mid_season_rank`? If the snapshot can produce ties,
  `compute_draft_order` needs a deterministic secondary key (use `regular_season_rank`, then team_id).
- **Traded picks**: draft order is by `original_owner_team_id` (a team's "natural" slot), but the pick
  is *made by* `current_owner_team_id` (§12.4 picks tradeable 2 yrs out). The slate must resolve each
  ordered slot to its `draft_pick` row and carry `current_owner_team_id` — a team can have 0 or many
  picks in a round. The lottery still uses the *non-playoff team's* standing for odds even if that
  team traded the pick away (the ball count belongs to the standings position, the resulting pick to
  the current owner).
- **Roster-room check on make-pick**: §7.3.2 requires an open roster slot. Since the draft happens
  preseason (offseason 32-man limit, `PRE_SEASON_CONTRACTS_PER_ROSTER_LIMIT`), confirm which limit
  applies during the draft window — likely the 32-man offseason limit, not the 6-RD regular-season
  limit. Clarify with roster-lock semantics (spec 07/roster_lock) before enforcing.
- **Pick acquired vs. used**: a passed pick still consumes its `order` slot (it becomes `Skipped`),
  matching importer behavior — the draft does not "snake back" to passed picks.
